# Contributing to tree-sitter-talkbank

This guide covers how to modify and test the CHAT grammar. Whether you're fixing a bug, adding support for a new CHAT construct, or improving editor queries — this is where to start.

## Prerequisites

- **Node.js** (20+) — for `tree-sitter-cli` tooling
- **tree-sitter-cli** — installed via `npm install` (devDependency)
- **Rust toolchain** (stable) — only needed for Rust binding tests

Optional (for upstream spec-driven workflows):
- Access to the repo root — for spec-driven test generation and symbol set updates

## Quick start

```bash
npm install                   # Install tree-sitter-cli and dependencies
npx tree-sitter generate      # Generate parser from grammar.js
npx tree-sitter test          # Run the test suite
npx tree-sitter parse file.cha  # Parse a CHAT file and inspect the tree
npm start                     # Launch the web playground
```

## Grammar modification workflow

1. **Edit `grammar.js`** — the single source of truth for the parser
2. **Generate the parser:**
   ```bash
   npx tree-sitter generate
   ```
   This produces `src/parser.c`, `src/grammar.json`, and `src/node-types.json`.
3. **Run the test suite:**
   ```bash
   npx tree-sitter test
   ```
4. **Parse real files** to check your change against real-world data:
   ```bash
   npx tree-sitter parse path/to/file.cha
   ```

If your change modifies node types (adding/removing/renaming rules), downstream consumers will need updates — see [Downstream impact](#downstream-impact) below.

## Generated files — what not to edit

| File | Generated by | Source |
|------|-------------|--------|
| `src/parser.c` | `tree-sitter generate` | `grammar.js` |
| `src/grammar.json` | `tree-sitter generate` | `grammar.js` |
| `src/node-types.json` | `tree-sitter generate` | `grammar.js` |
| `src/generated_symbol_sets.js` | `make symbols-gen` (from repo root) | `spec/symbols/` |
| `test/corpus/` | `make test-gen` (from repo root) | `spec/constructs/` and `spec/errors/` |

Never hand-edit any of these. If you need to change parser behavior, edit `grammar.js`. If you need to change test cases, read on.

## Testing

### Test corpus structure

The `test/corpus/` directory contains tree-sitter test files organized by category:

```
test/corpus/
├── errors/         # Error detection tests (generated from spec/errors/)
├── header/         # Header parsing tests
├── main_tier/      # Main tier / utterance tests
├── tiers/          # Dependent tier tests (MOR, GRA, PHO, etc.)
├── utterance/      # Complete utterance tests
└── word/           # Word-level parsing tests
```

Each `.txt` file uses tree-sitter's corpus format:

```
================
Test name
================

@UTF8
@Begin
*CHI:	hello .
@End

---

(document
  (utf8_header)
  (begin_header)
  (utterance
    (main_tier
      ...))
  (end_header))
```

### Spec-driven tests (generated)

Most test files are **generated** from specifications in the repository:

- **Construct specs** (`spec/constructs/`) define valid CHAT examples with expected parse trees
- **Error specs** (`spec/errors/`) define invalid CHAT examples with expected error codes

The generation pipeline (from the repo root):
```bash
make test-gen    # Runs gen_tree_sitter_tests → outputs to grammar/test/corpus/
```

This ensures specs are the single source of truth for what the grammar should accept/reject. Generated test files should not be edited by hand — change the spec and regenerate.

### Adding test cases manually

For grammar changes that aren't yet covered by specs, you can add test files directly to `test/corpus/`. Use descriptive names and follow the existing directory organization. If the test should become permanent, a corresponding spec should eventually be added to `spec/`.

### Running specific tests

```bash
npx tree-sitter test -f "test name"    # Filter by test name
npx tree-sitter test -f "header"       # Run all tests matching "header"
```

### Optional binding tests

Rust binding tests are the only downstream-critical binding checks for this repository right now.

```bash
# Rust binding
cargo test
```

## Editor queries

The `queries/` directory provides tree-sitter queries for editor integration:

| File | Purpose |
|------|---------|
| `highlights.scm` | Syntax highlighting |
| `injections.scm` | Language injection hooks |
| `locals.scm` | Local scope/query symbols |
| `tags.scm` | Symbol outline / navigation |

To test highlights in the playground:
```bash
npm start   # Opens tree-sitter playground with highlighting
```

When modifying grammar rules, check whether the queries need updates — especially `highlights.scm` if you add new node types.
Additional editor query families (for example folds/indents/textobjects) are post-v1 TODOs. For first release, prioritize grammar correctness and highlight stability.

## Consumer impact

If your grammar change adds, removes, or renames node types, update and retest any consumer that depends on those node names.

Changes that only affect internal parser states (precedence, conflict resolution) without changing the node type surface area are typically low-impact outside this repository.

## Symbol sets

`src/generated_symbol_sets.js` exports symbol constants used by `grammar.js`:

- `CA_DELIMITER_SYMBOLS` — CA delimiter characters
- `CA_ELEMENT_SYMBOLS` — CA element characters
- `EVENT_SEGMENT_FORBIDDEN_BASE` — characters that terminate event segments
- `EVENT_SEGMENT_FORBIDDEN_COMMON` — additional forbidden event characters

These are generated from `spec/symbols/symbol_registry.json`. To regenerate:

```bash
make symbols-gen    # from repo root
```

## Release process

1. Update version in `package.json`, `Cargo.toml`, `tree-sitter.json`, `Makefile`, and `CHANGELOG.md`
2. Ensure CI passes
3. Create a git tag: `git tag v0.X.Y`
4. Publish to npm: `npm publish`
5. Publish to crates.io: `cargo publish`

## Code style

- `grammar.js` uses JSDoc-style section comments with CHAT manual references
- Reference links follow the format: `https://talkbank.org/0info/manuals/CHAT.html#Main_Line`
- Prefer concrete construct anchors over broad aliases (for example `#File_Headers` over `#Headers`)
- Use `token()` for lexer-level rules, `prec()` for disambiguation
- Keep `extras: []` — CHAT whitespace is always explicit

## CHAT anchor validation

Anchor validation is run via `make chat-anchors-check` from the repo root.
